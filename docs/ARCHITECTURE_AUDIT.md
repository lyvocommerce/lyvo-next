5. Catalog & Data Flow

- Categories:
  - Fetched server-side in `app/layout.tsx` via `getAllCategories()` from `lib/categories.ts`.
  - Passed into `CategoriesProvider` to populate client context with `categories` and `categoriesTree`.
- Products:
  - Fetched client-side in `contexts/Products/ProductsProvider.tsx` from `https://fakestoreapi.com/products`.
  - Exposed via `ProductsContext` to UI components.
- Auth:
  - Client initializes `@twa-dev/sdk` in `TelegramAuthProvider` for Web App runtime.
  - When `initData` present but no direct user, client posts `initData` to `/api/auth/telegram` for server-side validation using `lib/telegram-auth.ts`.
- DB:
  - Prisma client (`lib/prisma.ts`) interfaces with Postgres; schema defines categories, products, merchants.

6. External Integrations

- Telegram Web App SDK: `@twa-dev/sdk` used dynamically on client.
- External product API: `https://fakestoreapi.com/products` for product data in dev/runtime.
- Database: Postgres via Prisma using `DATABASE_URL` env var.
- Bot secret: environment variables `TELEGRAM_BOT_SECRET`, `NEXT_PUBLIC_TELEGRAM_BOT_TOKEN`, or `TELEGRAM_BOT_TOKEN` used for validation.

7. Layer Violations (if any)

- Client-side product fetching uses a public external API (`fakestoreapi.com`), not the app's DB — this places data retrieval in client UI layer rather than a server/API layer.
- Theme application (`lib/telegram-theme.ts`) mutates `document` directly (expected on client). No code edits made; this is an observation of current structure.

8. Scalability Risks

- Client-side product fetch for full product list may expose high network usage and lacks server-side pagination or caching (observed behavior only).
- Prisma client configured as singleton in `lib/prisma.ts` (uses globalThis) — standard for Next.js, but actual connection pooling behavior depends on deployment and database settings (runtime unknown).
- Use of dynamic import for `@twa-dev/sdk` is client-safe but ties authentication flow to client runtime availability.

9. Technical Debt Indicators

- `lib/telegram-auth.ts` contains a code path that returns `true` (bypasses validation) if bot token env var is missing — indicates potential dev convenience that could be overlooked in production.
- Multiple environment variable names used for the same secret (`TELEGRAM_BOT_SECRET`, `NEXT_PUBLIC_TELEGRAM_BOT_TOKEN`, `TELEGRAM_BOT_TOKEN`) may cause configuration drift.
- External product data source (`fakestoreapi.com`) used in `ProductsProvider` suggests placeholder or mock integration rather than production data.
